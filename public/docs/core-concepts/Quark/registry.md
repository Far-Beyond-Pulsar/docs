The `Quark` registry is the central storage and execution engine for all commands in the system. It owns a `HashMap` of commands keyed by name, provides O(1) average-case lookup, and exposes a safe API for registration, execution, and introspection.

The registry is designed for **immutable-after-init** usage: commands are registered once at startup (or dynamically as plugins load) and then queried repeatedly at runtime. There is no support for un-registering or replacing commands — once registered, a command lives for the lifetime of the registry.

---

## 1. Internal Structure

```rust
pub struct Quark {
    commands: HashMap<String, Box<dyn Command>>,
}
```

The registry is a thin wrapper around a single data structure: a `HashMap<String, Box<dyn Command>>`.

### Key Type: `String`

Command names are stored as owned `String` rather than `&str` because:

1. Commands may be registered from various sources (macros, dynamic plugins, network messages) that don't share a single string lifetime.
2. The registry must own the keys to guarantee they live as long as the registry itself.
3. String comparison is fast — short command names (5–15 chars) hash and compare in single-digit nanoseconds on modern CPUs.

### Value Type: `Box<dyn Command>`

Commands are stored as boxed trait objects (`Box<dyn Command>`) rather than concrete types or generics. This enables:

1. **Heterogeneous storage** — Commands with different concrete types (`SpawnCommand`, `TeleportCommand`, etc.) can coexist in the same `HashMap`.
2. **No monomorphization bloat** — A generic `Quark<C: Command>` would generate a new copy of the registry code for every unique command set.
3. **Dynamic registration** — Commands can be added at runtime without knowing their types at compile time.

Each `Box<dyn Command>` is **16 bytes** on 64-bit systems:

- 8 bytes: pointer to the command struct (usually a zero-sized type generated by the macro)
- 8 bytes: pointer to the vtable (shared across all instances of the same command type)

For a registry with 100 commands, the total command storage is:

```
100 × 16 bytes = 1.6 KB (ignoring HashMap overhead)
```

## 2. Registration: `register_command()`

```rust
pub fn register_command<C: Command + 'static>(&mut self, command: C) {
    let name = command.name().to_string();
    self.commands.insert(name, Box::new(command));
}
```

### Signature Breakdown

- **`<C: Command + 'static>`** — The command type must implement `Command` and have no non-static references (i.e., it must be owned, not borrowed).
- **`&mut self`** — Registration requires mutable access to the registry.
- **`command: C`** — Takes ownership of the command by value (not by reference).

### Registration Steps

1. **Extract name** — Calls `command.name()` to get the command's string identifier.
2. **Heap-allocate** — Wraps the command in a `Box` to erase its concrete type.
3. **Insert into map** — Inserts the boxed command into the `HashMap` with the name as the key.

If a command with the same name already exists, it is **silently replaced**. This is intentional: the last-registered command wins. Use case: plugin B overrides a command from plugin A.

### Performance

Registration is a **one-time cost** paid at startup or plugin load time. For a typical command:

```rust
registry.register_command(SpawnCommand);
```

The cost is:

- **1 vtable lookup** — Calls `command.name()` via vtable (~2ns)
- **1 heap allocation** — `Box::new()` allocates 0–16 bytes depending on the command's size (~20ns)
- **1 hash map insertion** — Hashes the name string and inserts the entry (~50ns for a 10-char string)

**Total: ~70–100ns per command.**

For 1000 commands, total registration time is ~100μs — imperceptible.

### Zero-Sized Types Optimization

Most macro-generated commands are **zero-sized types (ZSTs)**:

```rust
struct SpawnCommand;  // sizeof(SpawnCommand) == 0

impl Command for SpawnCommand { /* ... */ }
```

When you call `Box::new(SpawnCommand)`, the compiler recognizes that the type is zero-sized and optimizes the allocation to a **single pointer** (no actual heap allocation occurs — the pointer is just a tagged address). The command still behaves as a regular `Box<dyn Command>`, but with zero runtime overhead.

## 3. Execution: `run()` and `run_async()`

### 3.1 Synchronous Execution: `run()`

```rust
pub fn run(&self, input: &str) -> Result<()> {
    let (name, args) = parse_command_string(input)?;

    let command = self.commands
        .get(&name)
        .ok_or_else(|| CommandError::NotFound(name.clone()))?;

    if command.is_async() {
        return Err(CommandError::RequiresAsyncRuntime);
    }

    command.execute(args)
}
```

**Execution steps:**

1. **Parse input** — Calls `parse_command_string()` to split the input into a command name and argument list (see [Parser documentation](./parser)).
2. **Lookup command** — Queries the `HashMap` for the command by name. If not found, returns `CommandError::NotFound`.
3. **Check async flag** — Verifies the command is synchronous. If `is_async() == true`, returns an error.
4. **Execute** — Calls `command.execute(args)` via vtable and returns the result.

**Performance:**

- **Parse:** O(N) where N = input length (~100ns for typical 20-char input)
- **Lookup:** O(1) average case (~10ns for hash + comparison)
- **Vtable call:** ~2ns per virtual method call
- **Execution:** Varies by command (user-defined)

**Total overhead: ~112ns + command execution time.**

The parser allocates a `Vec<String>` for the arguments, which is the only heap allocation in the hot path. For a command with 3 arguments, this is ~48 bytes.

### 3.2 Asynchronous Execution: `run_async()`

```rust
pub async fn run_async(&self, input: &str) -> Result<()> {
    let (name, args) = parse_command_string(input)?;

    let command = self.commands
        .get(&name)
        .ok_or_else(|| CommandError::NotFound(name.clone()))?;

    command.execute_async(args).await
}
```

The async version differs in two ways:

1. **No async check** — Both sync and async commands can be called via `run_async()`. Sync commands use the default `execute_async()` implementation that wraps the sync result in a ready future.
2. **`.await`** — The returned future is awaited before returning.

**Performance:**

- **Parse:** Same as `run()` (~100ns)
- **Lookup:** Same as `run()` (~10ns)
- **Vtable call:** ~2ns
- **Async overhead:** ~10ns for the future allocation (if not optimized away by the compiler)
- **Execution:** Async work defined by the command

For **sync commands** called via `run_async()`, the future completes immediately (no suspension, no task spawning). The compiler often optimizes this to be identical to a direct `execute()` call.

For **async commands**, the future may suspend multiple times, yielding to the executor. The registry does not spawn tasks — it simply awaits the future inline, so the caller controls the execution context.

## 4. Introspection: `list()` and `get_docs()`

### 4.1 `list() -> Vec<&dyn Command>`

Returns a vector of references to all registered commands:

```rust
pub fn list(&self) -> Vec<&dyn Command> {
    self.commands.values().map(|cmd| cmd.as_ref()).collect()
}
```

**Use cases:**

- Generating help menus (`for cmd in registry.list() { println!("{}: {}", cmd.name(), cmd.short()); }`)
- Autocomplete suggestions (filter commands by prefix)
- Debug inspection (dump all registered commands to a log file)

**Performance:**

- O(N) where N = number of commands
- Allocates a `Vec<&dyn Command>` with capacity N (~8N bytes)
- For 100 commands: ~800 bytes + iteration overhead (~1μs)

The returned references are valid as long as the registry exists (the commands are owned by the registry, so the references have the same lifetime as `&self`).

### 4.2 `get_docs(name: &str) -> Option<&str>`

Returns the documentation for a specific command by name:

```rust
pub fn get_docs(&self, name: &str) -> Option<&str> {
    self.commands.get(name).map(|cmd| cmd.docs())
}
```

**Use cases:**

- Implementing a `help <command>` command that shows detailed docs
- Context-sensitive tooltips in graphical consoles
- API documentation generators

**Performance:**

- O(1) average case (hash map lookup + vtable call)
- ~12ns total

If the command does not exist, returns `None`.

## 5. Thread Safety

The `Quark` struct is **not thread-safe for mutation** but **is thread-safe for concurrent reads**.

### Read-Only Access: `&Quark` is `Sync`

Multiple threads can hold `&Quark` references and call `run()`, `run_async()`, `list()`, or `get_docs()` concurrently:

```rust
use std::sync::Arc;
use std::thread;

let registry = Arc::new(registry);

let r1 = Arc::clone(&registry);
let h1 = thread::spawn(move || {
    r1.run("command1").unwrap();
});

let r2 = Arc::clone(&registry);
let h2 = thread::spawn(move || {
    r2.run("command2").unwrap();
});

h1.join().unwrap();
h2.join().unwrap();
```

This works because:

1. `HashMap` is `Sync` for immutable access (no internal mutability).
2. All `Command` implementations must be `Sync` (enforced by the `Command: Send + Sync` trait bound).

### Mutation: `&mut Quark` is NOT `Sync`

The `register_command()` method requires `&mut self`, so it cannot be called concurrently. If you need to register commands from multiple threads, wrap the registry in a `Mutex<Quark>`:

```rust
use std::sync::{Arc, Mutex};

let registry = Arc::new(Mutex::new(Quark::new()));

let r1 = Arc::clone(&registry);
thread::spawn(move || {
    let mut reg = r1.lock().unwrap();
    reg.register_command(Command1);
});

let r2 = Arc::clone(&registry);
thread::spawn(move || {
    let mut reg = r2.lock().unwrap();
    reg.register_command(Command2);
});
```

In practice, commands are almost always registered single-threaded at startup, so the `Mutex` overhead is rarely needed.

## 6. HashMap Performance Characteristics

### 6.1 Hash Function

The `HashMap` uses the default Rust hasher (`std::collections::hash_map::DefaultHasher`), which is **SipHash-1-3** — a cryptographically secure hash function designed to resist hash-flooding DoS attacks.

**Pros:**

- Collision-resistant even with adversarial input
- Consistent performance across different input patterns
- No need to worry about malicious command names

**Cons:**

- Slower than non-cryptographic hash functions like FxHash (~10–20ns vs ~2–5ns per hash)

For command registries with < 10,000 commands, the difference is negligible. If profiling shows `HashMap::get()` as a bottleneck, the internal map can be replaced with `FxHashMap` from the `rustc-hash` crate:

```rust
use rustc_hash::FxHashMap;

pub struct Quark {
    commands: FxHashMap<String, Box<dyn Command>>,
}
```

This requires no API changes — `FxHashMap` is a drop-in replacement.

### 6.2 Collision Handling

Rust's `HashMap` uses **open addressing with quadratic probing** (as of Rust 1.70+). When two keys hash to the same slot, the map searches for the next available slot using a quadratic sequence:

```
slot = (hash + 0^2) % capacity
slot = (hash + 1^2) % capacity
slot = (hash + 2^2) % capacity
...
```

This avoids clustering (a problem with linear probing) and provides O(1) expected lookup time even under moderate load.

### 6.3 Load Factor and Resizing

The `HashMap` maintains a **load factor of ~0.875** (87.5% full). When the number of entries exceeds `capacity × 0.875`, the map doubles its capacity and rehashes all entries.

For a registry with 100 commands:

- Initial capacity: typically 128 (next power-of-two above 100 / 0.875 ≈ 114)
- Memory usage: `128 × sizeof(Entry)` where `Entry ≈ 40 bytes` (key, value, hash)
- Total: ~5 KB

Resizing happens at registration time, not at execution time, so there is no runtime overhead for lookups.

### 6.4 Lookup Time Complexity

| Operation | Average Case | Worst Case |
|-----------|--------------|------------|
| `get()` | O(1) | O(N) (degenerate collisions) |
| `insert()` | O(1) amortized | O(N) (on resize) |
| `len()` | O(1) | O(1) |

The worst case only occurs if an adversary crafts command names specifically to collide under the hash function. SipHash makes this computationally infeasible.

## 7. Memory Layout

For a registry with N commands, each of size S:

```
Total memory = HashMap overhead + N × (Entry size)

Entry size = sizeof(String) + sizeof(Box<dyn Command>) + sizeof(u64)
           = 24 + 16 + 8
           = 48 bytes

HashMap overhead ≈ capacity × 8 bytes (hash table slots)
                  ≈ (N / 0.875) × 8
                  ≈ N × 9.14 bytes
```

**Example: 100 commands**

```
Entry storage:   100 × 48 = 4,800 bytes
Hash table:      100 × 9.14 ≈ 914 bytes
Total:           ~5.7 KB
```

For comparison, a single 4K texture is 12 MB. The registry memory footprint is negligible.

## 8. Error Handling

All public methods return `Result<T, CommandError>` (or `Result<T>` where `Result<T> = Result<T, CommandError>`).

### 8.1 `CommandError::NotFound`

Returned by `run()` and `run_async()` when the command name does not exist:

```rust
registry.run("nonexistent")?;
// Error: CommandError::NotFound("nonexistent")
```

### 8.2 `CommandError::RequiresAsyncRuntime`

Returned by `run()` when called on an async command:

```rust
registry.run("async_command")?;
// Error: CommandError::RequiresAsyncRuntime
```

### 8.3 Other Errors

All other errors (`ArgumentCountMismatch`, `TypeConversionError`, `ExecutionError`) are propagated from the command's `execute()` or `execute_async()` implementation.

## 9. Comparison: Quark vs Alternatives

### 9.1 String-Based Dispatch (e.g., `match` on `input.split()`)

```rust
// Without Quark
match input.split_whitespace().next() {
    Some("spawn") => handle_spawn(input),
    Some("teleport") => handle_teleport(input),
    _ => println!("Unknown command"),
}
```

**Problems:**

- No extensibility — adding a command requires modifying the central `match` statement
- No documentation — no way to list available commands or get help text
- No type safety — each handler manually parses arguments
- No async support — all handlers must be sync or manually spawned

### 9.2 Function Pointer Registry (e.g., `HashMap<String, fn(&str)>`)

```rust
// Function pointer approach
let mut registry: HashMap<String, fn(&str)> = HashMap::new();
registry.insert("spawn".to_string(), handle_spawn);
registry.insert("teleport".to_string(), handle_teleport);
```

**Problems:**

- No metadata — no way to get command syntax or documentation
- All handlers must have the same signature `fn(&str)` — cannot vary by argument count or type
- No async support — function pointers cannot be `async fn`

### 9.3 Quark

- **Extensible:** Commands can be registered from any crate, no central match statement
- **Documented:** Every command has `name()`, `syntax()`, `short()`, `docs()`
- **Type-safe:** Arguments are validated at compile time by the macro
- **Async-ready:** Seamless support for both sync and async commands
- **Zero runtime overhead:** No dynamic dispatch beyond the vtable call (which is ~2ns)

## 10. Advanced: Custom Registry Wrappers

The `Quark` struct can be wrapped to add custom behavior:

### 10.1 Middleware Registry

```rust
struct MiddlewareRegistry {
    inner: Quark,
}

impl MiddlewareRegistry {
    pub fn run(&self, input: &str) -> Result<()> {
        println!("Before: {}", input);
        let result = self.inner.run(input);
        match &result {
            Ok(()) => println!("After: Success"),
            Err(e) => println!("After: Error {}", e),
        }
        result
    }
}
```

### 10.2 Permission-Checking Registry

```rust
struct SecureRegistry {
    inner: Quark,
    permissions: HashMap<String, Vec<String>>,  // command -> allowed_users
}

impl SecureRegistry {
    pub fn run(&self, input: &str, user: &str) -> Result<()> {
        let (name, _) = parse_command_string(input)?;

        if let Some(allowed) = self.permissions.get(&name) {
            if !allowed.contains(&user.to_string()) {
                return Err(CommandError::ExecutionError(
                    format!("User {} not authorized", user)
                ));
            }
        }

        self.inner.run(input)
    }
}
```

### 10.3 Rate-Limiting Registry

```rust
use std::collections::HashMap;
use std::time::{Duration, Instant};

struct RateLimitedRegistry {
    inner: Quark,
    last_call: HashMap<String, Instant>,
    cooldown: Duration,
}

impl RateLimitedRegistry {
    pub fn run(&mut self, input: &str, user: &str) -> Result<()> {
        let key = format!("{}:{}", user, input);

        if let Some(&last) = self.last_call.get(&key) {
            if last.elapsed() < self.cooldown {
                return Err(CommandError::ExecutionError(
                    "Command on cooldown".to_string()
                ));
            }
        }

        self.last_call.insert(key, Instant::now());
        self.inner.run(input)
    }
}
```

## 11. Design Rationale

### Why Not a Static Global Registry?

A common pattern in C/C++ is a global command registry that auto-registers commands at static init time:

```cpp
// C++ approach
REGISTER_COMMAND("spawn", handle_spawn);  // Macro registers at static init
```

Rust discourages global mutable state, and static initialization order is undefined. Quark instead uses explicit registration:

```rust
let mut registry = Quark::new();
registry.register_command(SpawnCommand);
```

This gives the caller full control over registration order and allows multiple independent registries to coexist (useful for testing or sandboxing).

### Why `&str` Input Instead of Pre-Parsed?

The `run()` method takes `&str` instead of a pre-parsed `(String, Vec<String>)` to hide the parser implementation from the caller. If the parser changes (e.g., to support escape sequences or quoted strings), the API remains stable.

Callers who have already parsed the input can directly call `command.execute(args)` after looking up the command themselves.

### Why Return `Result<()>` Instead of `Result<T>`?

Commands return `Result<()>` (success/failure) rather than a generic `Result<T>` (success with value). This is because:

1. Commands are side-effectful operations (spawn entity, teleport player, save file) — they don't produce return values.
2. A generic return type would require the registry to store `Box<dyn Command<Output = T>>` for every possible T, which is impossible.

Commands that need to return values can do so via shared state (e.g., writing to a channel, updating a global result buffer, or using a closure-based design).
