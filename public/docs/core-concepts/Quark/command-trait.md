The `Command` trait is the foundational abstraction in Quark. Every command registered in the system implements this trait, which defines both the command's metadata (name, syntax, documentation) and its execution behavior (sync vs async, argument handling).

The trait is deliberately minimal — only 7 methods, 5 of which are pure metadata accessors. This simplicity enables maximum flexibility: commands can be generated by macros, implemented manually, or even generated dynamically at runtime from external data sources.

---

## 1. Trait Definition

```rust
pub trait Command: Send + Sync {
    fn name(&self) -> &str;
    fn syntax(&self) -> &str;
    fn short(&self) -> &str;
    fn docs(&self) -> &str;
    fn is_async(&self) -> bool { false }
    fn execute(&self, args: Vec<String>) -> Result<()> {
        Err(CommandError::RequiresAsyncRuntime)
    }
    fn execute_async<'a>(
        &'a self,
        args: Vec<String>,
    ) -> Pin<Box<dyn Future<Output = Result<()>> + Send + 'a>> {
        let result = self.execute(args);
        Box::pin(async move { result })
    }
}
```

The trait has seven methods total:

- **Four metadata accessors** — `name()`, `syntax()`, `short()`, `docs()`
- **One async flag** — `is_async()`
- **Two execution methods** — `execute()`, `execute_async()`

### The `Send + Sync` Bound

All commands must be `Send + Sync`, meaning they can be safely transferred between threads and shared concurrently. This requirement exists because:

1. The `Quark` registry stores commands as `Box<dyn Command>`, which must be thread-safe to allow the registry to be used across thread boundaries.
2. Commands may be invoked from multiple threads simultaneously (e.g., networked multiplayer server handling concurrent client commands).
3. Async commands may be spawned onto a thread pool executor, requiring `Send`.

In practice, this is rarely a constraint — most commands operate on thread-safe types (`Arc`, `Mutex`, message channels) or are stateless zero-sized structs generated by the `#[command]` macro.

## 2. Metadata Methods

### 2.1 `name(&self) -> &str`

Returns the command's unique identifier. This is the string users type to invoke the command:

```rust
cmd.name() // "spawn", "teleport", "help", etc.
```

Command names are case-sensitive and must be valid UTF-8 strings. The registry uses this value as the key in the internal `HashMap<String, Box<dyn Command>>`.

**Naming conventions:**

- Lowercase, snake_case preferred: `spawn_enemy`, `set_volume`
- Hyphens allowed for multi-word commands: `toggle-debug-draw`
- No whitespace: `teleport player` is invalid (would be parsed as two arguments)

### 2.2 `syntax(&self) -> &str`

Returns a human-readable syntax string showing how to use the command. This typically includes the command name followed by placeholders for arguments:

```rust
cmd.syntax() // "spawn <entity> <count>", "teleport <x> <y> <z>"
```

**Formatting conventions:**

- Required arguments in angle brackets: `<entity>`
- Optional arguments in square brackets: `[player_id]`
- Multiple choice with pipe: `(on|off)`
- Variable arguments with ellipsis: `<args...>`

This string is purely documentation — it does not affect parsing or validation. The actual argument types are enforced by the `execute()` implementation.

### 2.3 `short(&self) -> &str`

Returns a one-line description of what the command does. This should be a concise imperative phrase, typically under 80 characters:

```rust
cmd.short() // "Spawn entities into the world", "Teleport the player"
```

This text is shown in command listings (e.g., `help` command output) where space is limited.

### 2.4 `docs(&self) -> &str`

Returns detailed documentation for the command. This can be multiple paragraphs and should include:

- What the command does
- What each argument represents
- Example usage
- Side effects or caveats

```rust
cmd.docs()
// "Spawns the specified number of entities at the player's current location.
//
//  Arguments:
//    <entity> - Entity type name (e.g., 'goblin', 'chest')
//    <count>  - Number of entities to spawn (positive integer)
//
//  Example: spawn goblin 5
//
//  Note: Spawned entities are placed in a grid pattern around the player."
```

This text is shown when the user requests detailed help for a specific command (e.g., `help spawn`).

## 3. Execution Methods

### 3.1 `is_async(&self) -> bool`

Returns `true` if the command is asynchronous and must be executed via `run_async()`. The default implementation returns `false`, indicating a synchronous command.

Async commands override this method to return `true`:

```rust
fn is_async(&self) -> bool {
    true
}
```

The registry uses this flag to enforce correct usage:

- If `run()` is called on an async command (`is_async() == true`), it returns `CommandError::RequiresAsyncRuntime`.
- If `run_async()` is called on a sync command (`is_async() == false`), the default `execute_async()` wrapper is used (see Section 3.3).

### 3.2 `execute(&self, args: Vec<String>) -> Result<()>`

Executes the command synchronously with the provided arguments. This method is called by `Quark::run()`.

**Arguments:**

- `args` — A vector of argument strings parsed from the command input. The first element is the first argument **after** the command name (the command name itself is not included).

**Example:**

User types: `"spawn goblin 5"`

Parsed as:
- Command name: `"spawn"`
- Args passed to `execute()`: `vec!["goblin", "5"]`

**Returns:**

- `Ok(())` on success
- `Err(CommandError)` on failure (wrong argument count, type conversion failure, execution error)

**Default implementation:**

The trait provides a default implementation that returns `RequiresAsyncRuntime`. This allows async-only commands to omit the `execute()` implementation — they only need to implement `execute_async()`.

**Typical implementation pattern (generated by `#[command]` macro):**

```rust
fn execute(&self, args: Vec<String>) -> Result<()> {
    // 1. Validate argument count
    if args.len() != 2 {
        return Err(CommandError::ArgumentCountMismatch {
            expected: 2,
            got: args.len(),
        });
    }

    // 2. Parse/convert arguments
    let entity = args[0].clone();
    let count: usize = args[1].parse()
        .map_err(|_| CommandError::TypeConversionError {
            arg: args[1].clone(),
            target_type: "usize",
        })?;

    // 3. Call the actual command function
    spawn(entity, count);

    Ok(())
}
```

The macro automates this boilerplate for every command, generating the correct parsing logic based on the function's type signature.

### 3.3 `execute_async<'a>(...) -> Pin<Box<dyn Future<...>>>`

Executes the command asynchronously, returning a pinned boxed future. This method is called by `Quark::run_async()`.

**Signature breakdown:**

```rust
fn execute_async<'a>(
    &'a self,
    args: Vec<String>,
) -> Pin<Box<dyn Future<Output = Result<()>> + Send + 'a>>
```

- **Lifetime `'a`** — The future borrows `&self`, so it cannot outlive the command object.
- **`Pin<Box<...>>`** — The future is heap-allocated and pinned, allowing it to be polled by any async runtime.
- **`+ Send`** — The future must be `Send` to support multi-threaded executors (Tokio, async-std).
- **`Output = Result<()>`** — The future resolves to the same `Result` type as `execute()`.

**Default implementation:**

The trait provides a default implementation that wraps the synchronous `execute()` call in a ready future:

```rust
fn execute_async<'a>(
    &'a self,
    args: Vec<String>,
) -> Pin<Box<dyn Future<Output = Result<()>> + Send + 'a>> {
    let result = self.execute(args);
    Box::pin(async move { result })
}
```

This allows synchronous commands to be called through `run_async()` without any special handling — the sync work happens immediately, and the future completes on the first poll.

**Async command implementation pattern:**

```rust
fn execute_async<'a>(
    &'a self,
    args: Vec<String>,
) -> Pin<Box<dyn Future<Output = Result<()>> + Send + 'a>> {
    Box::pin(async move {
        // Validate args (synchronous)
        if args.len() != 1 {
            return Err(CommandError::ArgumentCountMismatch {
                expected: 1,
                got: args.len(),
            });
        }

        // Async work
        let filename = &args[0];
        tokio::fs::write(filename, "data").await
            .map_err(|e| CommandError::ExecutionError(e.to_string()))?;

        Ok(())
    })
}
```

Note that argument validation happens inside the `async move` block but is still synchronous — only the actual I/O or other async operations use `.await`.

## 4. Manual Implementation Example

While most commands are defined via the `#[command]` macro, manual implementation is fully supported. This is useful for:

- Commands with complex validation logic
- Commands that need custom error messages
- Commands generated dynamically from external data

```rust
use quark::{Command, CommandError, Result};
use std::pin::Pin;
use std::future::Future;

struct CustomCommand;

impl Command for CustomCommand {
    fn name(&self) -> &str {
        "custom"
    }

    fn syntax(&self) -> &str {
        "custom <arg1> <arg2>"
    }

    fn short(&self) -> &str {
        "A custom command implementation"
    }

    fn docs(&self) -> &str {
        "This command demonstrates manual Command trait implementation.\n\
         It takes two arguments and prints them."
    }

    fn execute(&self, args: Vec<String>) -> Result<()> {
        if args.len() != 2 {
            return Err(CommandError::ArgumentCountMismatch {
                expected: 2,
                got: args.len(),
            });
        }

        println!("Arg1: {}, Arg2: {}", args[0], args[1]);
        Ok(())
    }
}

// Usage:
let mut registry = Quark::new();
registry.register_command(CustomCommand);
registry.run("custom hello world").unwrap();
```

## 5. Command Trait Object Size

Commands are stored in the registry as `Box<dyn Command>`. The size of a trait object is:

```
size_of::<Box<dyn Command>>() = 16 bytes on 64-bit systems
```

This is two pointers:

1. **Data pointer** — Points to the concrete command struct
2. **Vtable pointer** — Points to the trait's virtual method table

The concrete command struct itself is typically **zero-sized** for macro-generated commands, since they contain no state — only method implementations. Example:

```rust
struct SpawnCommand;  // Zero-sized type (ZST)

impl Command for SpawnCommand { /* ... */ }

size_of::<SpawnCommand>() == 0
size_of::<Box<SpawnCommand>>() == 8  // Just the pointer
size_of::<Box<dyn Command>>() == 16  // Pointer + vtable pointer
```

For a registry with 100 commands, the total vtable overhead is:

```
100 commands × 16 bytes = 1.6 KB
```

This is negligible. The bulk of memory usage comes from the `HashMap` internal structure (nodes, hash table slots), which scales as O(N) with the number of commands.

## 6. Thread Safety

The `Send + Sync` bounds guarantee that:

1. **`Send`** — A command can be transferred to another thread (via `send()` on a channel, `spawn()` on a thread pool, etc.).
2. **`Sync`** — A shared reference `&Command` can be accessed from multiple threads concurrently.

This enables concurrent command execution:

```rust
use std::sync::Arc;
use std::thread;

let registry = Arc::new(registry);

let r1 = Arc::clone(&registry);
let h1 = thread::spawn(move || {
    r1.run("command1").unwrap();
});

let r2 = Arc::clone(&registry);
let h2 = thread::spawn(move || {
    r2.run("command2").unwrap();
});

h1.join().unwrap();
h2.join().unwrap();
```

Each thread can call `run()` concurrently because `&Quark` is `Sync` (the internal `HashMap` is never mutated after initialization, and all commands are `Sync`).

## 7. Error Handling

All execution methods return `Result<(), CommandError>`. The `CommandError` enum covers all failure modes:

```rust
pub enum CommandError {
    NotFound(String),
    ArgumentCountMismatch { expected: usize, got: usize },
    TypeConversionError { arg: String, target_type: &'static str },
    RequiresAsyncRuntime,
    ExecutionError(String),
}
```

### 7.1 `NotFound`

Returned by the registry (not by `execute()`) when the command name does not exist in the registry.

### 7.2 `ArgumentCountMismatch`

Returned when the provided argument count does not match the expected count:

```rust
// Command expects 2 args, user provides 1
Err(CommandError::ArgumentCountMismatch { expected: 2, got: 1 })
```

### 7.3 `TypeConversionError`

Returned when an argument cannot be parsed as the expected type:

```rust
// User provides "abc" for a usize parameter
Err(CommandError::TypeConversionError {
    arg: "abc".to_string(),
    target_type: "usize",
})
```

### 7.4 `RequiresAsyncRuntime`

Returned when `run()` is called on an async command, or when a sync command has no `execute()` implementation.

### 7.5 `ExecutionError`

Generic catch-all for command-specific errors:

```rust
Err(CommandError::ExecutionError("File not found: world.dat".to_string()))
```

This is typically used to wrap underlying errors from I/O, network, or other subsystems.

## 8. Comparison: Macro-Generated vs Manual

| Aspect | Macro-Generated | Manual Implementation |
|--------|-----------------|----------------------|
| **Boilerplate** | Zero — macro handles all parsing | Must write argument validation and conversion manually |
| **Type safety** | Compile-time checked function signature | Runtime checks only |
| **Flexibility** | Limited to function arguments | Full control over execution logic |
| **Error messages** | Automatic, consistent | Custom per command |
| **Async support** | Automatic detection of `async fn` | Must implement `is_async()` and `execute_async()` manually |
| **Code size** | ~50 lines of generated code per command | ~20–30 lines of hand-written code per command |

**Recommendation:** Use the `#[command]` macro for 95% of commands. Reserve manual implementation for edge cases where the macro's limitations are prohibitive (e.g., variable-length argument lists, complex validation logic, dynamic command generation).

## 9. Advanced: Command Composition

Commands can wrap other commands to implement middleware-like behavior:

```rust
struct LoggedCommand<C: Command> {
    inner: C,
}

impl<C: Command> Command for LoggedCommand<C> {
    fn name(&self) -> &str {
        self.inner.name()
    }

    fn syntax(&self) -> &str {
        self.inner.syntax()
    }

    fn short(&self) -> &str {
        self.inner.short()
    }

    fn docs(&self) -> &str {
        self.inner.docs()
    }

    fn execute(&self, args: Vec<String>) -> Result<()> {
        println!("Executing: {} {:?}", self.name(), args);
        let result = self.inner.execute(args);
        match &result {
            Ok(()) => println!("Success"),
            Err(e) => println!("Error: {}", e),
        }
        result
    }
}

// Wrap an existing command:
registry.register_command(LoggedCommand { inner: SpawnCommand });
```

This pattern enables:

- Logging and metrics collection
- Permission checks (reject execution if user lacks privileges)
- Rate limiting (throttle command execution per user)
- Audit trails (record who executed what when)

## 10. Design Rationale

### Why Two Execution Methods?

Rust's async functions return opaque `impl Future` types that cannot be erased into a trait object directly. The only way to store async functions in a homogeneous collection (like a `HashMap`) is to box them:

```rust
Pin<Box<dyn Future<Output = Result<()>> + Send>>
```

This requires `execute_async()` to return a boxed future, not an `impl Future`.

Sync commands don't need boxing — they can return `Result<()>` directly. Having two methods (`execute` for sync, `execute_async` for async) avoids forcing sync commands to allocate a `Box` they don't need.

### Why Not Variadic Generics?

Rust does not have variadic generics (as of 2024), so there is no way to write a generic `fn execute<T1, T2, ...>(&self, args: (T1, T2, ...))` that works for arbitrary argument counts. The macro works around this by generating a separate parsing function for each command's specific argument types.

### Why `Vec<String>` Instead of `&[&str]`?

The parser allocates `String` for each argument to support quoted strings and escape sequences:

```
Input: "say \"hello world\""
Args: ["say", "hello world"]  // The quotes are removed, so we need an owned String
```

Using `&[&str]` would require the arguments to be substrings of the original input, but escaping and quote removal produce new strings that don't exist in the original buffer.

The performance cost is negligible (< 1μs per command invocation for typical argument counts).
